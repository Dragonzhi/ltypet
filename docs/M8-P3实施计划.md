## 1. 结论与实施门

P2 的 GUI 导出、文件落盘和重新导入已经人工验证通过，可以进入 P3。

P3 的目标是让生产桌宠通过现有 `motion.play` 语义协议播放编辑器导出的 v1 Clip，而不是在 `TianyiPet` 中再增加一套直接操作 SVG 的动作代码。完成后，单击、Mock Agent 和未来 LLM 仍只提交：

```ts
motion.play("wave")
```

调度器、执行器、`SvgCharacterRenderer`、运行时动作播放器和 SVG Part rig 依次完成校验、播放、中断与恢复。

开始时间轴播放器之前必须先通过 P3-0：

1. 生产 SVG 改为 rig 指纹对应的 `小洛宝.glax.svg`，或重新生成一份与生产 SVG 精确匹配的 rig；不得忽略指纹不一致。
2. 仓库内固化一份通过编辑器导出的正式 motions，至少包含 `id: "wave"`、`loop: "none"`。
3. 根项目建立对 `@ltypet/character-motion` 公共 API 的明确构建解析方式，不从包内部路径导入。
4. 在新动作通过真实窗口验收前保留旧 CSS 挥手作为对照；通过后删除旧 `pet-wave` 和前景手臂复制方案，不能长期保留两套生产事实。

## 2. 当前事实与差距

### 2.1 已具备

- M1-M4 已有 `motion.play → BehaviorScheduler → PetActionExecutor → CharacterRenderer` 纵向链路。
- `SvgCharacterRenderer` 已封装能力查询、表情、视线、reset 和 dispose。
- `@ltypet/character-motion` 已提供 v1 rig/motions 校验、帧换算、缓动、Clip 采样、pivot 矩阵和稳定序列化。
- `TianyiArtwork` 已建立手臂、头部、马尾、刘海、鬓发、发饰和耳朵的程序动画包装层。
- 拖动、点击穿透、窗口移动、Mock Agent、调度中断和结构化事件已有测试基线。

### 2.2 必须消除的双重事实

| 当前事实 | P3 目标 |
|---|---|
| 生产 `TianyiArtwork` 导入 `小洛宝.svg` | 使用 rig 指纹绑定的 `小洛宝.glax.svg` |
| `SvgCharacterRenderer` 只认识硬编码 `SUPPORTED_MOTIONS = ["wave"]` | capabilities 来自已校验 MotionLibrary |
| 挥手由 `data-action` + CSS `@keyframes pet-wave` 播放 | 挥手由 v1 Clip 和 runtime RAF 播放 |
| 右手通过复制前景手臂切换层级 | `renderSlot` 离散重排真实 Part wrapper |
| 中断只清 listener/timer，底层 Promise 不结算 | 完成、中断、失败和 dispose 都只结算一次并完整恢复 |
| 程序动画直接写 source Part 的 CSS transform | authored、interaction、procedural 使用独立包装层 |
| 当前示例动作名为 `p0-wave` 且 `loop: repeat` | 生产动作名为 `wave` 且单击后可以自然完成 |

## 3. 范围

### 3.1 必须完成

- 加载并严格校验一套生产 artwork、rig 和 motions。
- 从动作库动态生成 renderer motion capabilities。
- 用时间戳和 `requestAnimationFrame` 播放 v1 Clip，支持 fps、speed、缓动、首尾帧和 repeat。
- 每帧把 authored transform 与 opacity 投影到独立 Part wrapper。
- 在不重挂载 SVG 的情况下执行离散 `renderSlot`。
- 正确处理完成、中断、超时、reset、dispose、StrictMode 和 HMR。
- 根据 `suppressProceduralChannels` 暂停并恢复呼吸、随机眨眼、视线、头发物理和耳朵。
- 支持白名单动作事件；未支持事件不得执行任意脚本或外部资源。
- `prefers-reduced-motion` 下可靠完成动作 Promise，不遗留姿态或 suppression。
- Mock Agent、单击招手和键盘招手都走同一命名动作。
- 删除通过验收后的旧 CSS 挥手和前景手臂 clone。

### 3.2 明确不做

- 不把 svgcanvas、编辑器 React 组件、session、draft 或 history 引入生产 bundle。
- 不修改 v1 schema，不新增 P3 私有动作字段。
- 不做两套服装、动作重定向、fallback motion map 或 outfit 切换；这些属于 P4。
- 不做骨骼 IK、蒙皮、网格、路径变形或混合树。
- 不在本阶段增加音频系统；`sfx` 事件只产生受控诊断，不加载 URL。
- 不把 idle 改造成一个永不结束的 scheduler body-motion；持续待机仍由程序动画负责。
- 不增加用户设置项或新的 Tauri capability。

## 4. 生产资产与依赖

### 4.1 单一资产目录

正式运行时资产统一放置为：

```text
src/assets/character/xiaoluobao/
  artwork.svg          # 来自当前权威 小洛宝.glax.svg
  rig.v1.json          # 编辑器/生成器导出的 canonical rig
  motions.v1.json      # 编辑器导出的 canonical motions
```

约束：

- `rig.artwork.source`、fingerprint 和 viewBox 必须与 `artwork.svg` 一致。
- rig 生成脚本改为写入上述唯一正式文件；共享核心测试和编辑器示例都读取它，不再维护第二份“正式 fixture”。
- motions 至少包含非循环 `wave`。若用户验收文件含 `idle`、`bow` 等 Clip，可以一起进入动作库，但 P3 只承诺验证 `wave`。
- 资产更新必须经过 canonical 校验测试，禁止手工改 fingerprint 绕过检查。

### 4.2 根项目消费共享核心

根 Vite 和 TypeScript 使用同一个 `@ltypet/character-motion` 公共入口。实施时二选一并记录最终选择：

1. 推荐：在根 `vite.config.ts` 与 `tsconfig.json` 中增加指向 `packages/character-motion/src` 的一致 alias，并在根依赖中显式提供共享核心结构校验所需的 `ajv@8.20.0`。
2. 若改为本地 package dependency，则根 scripts 必须可靠地先构建共享包，不能依赖某台机器残留的 `packages/character-motion/dist`。

不得使用 `../../packages/character-motion/src/...` 深层相对导入，也不得复制采样或矩阵代码到 `src/`。

## 5. 运行时架构

```text
versioned artwork / rig / motions
               │
               ▼
   loadCharacterMotionBundle
   schema + semantics + fingerprint
               │
               ▼
        SvgRuntimeRig (DOM)
 Part binding / wrappers / slots / restore
               │
               ▼
        SvgMotionPlayer (RAF)
 frame / events / suppression / lifecycle
               │
               ▼
       SvgCharacterRenderer
               │
               ▼
 PetActionExecutor → BehaviorScheduler
               ▲
     click / keyboard / Mock Agent
```

建议目录：

```text
src/motion/runtime/
  loadCharacterMotionBundle.ts
  playbackMath.ts
  motionEvents.ts
  SvgRuntimeRig.ts
  SvgMotionPlayer.ts
  types.ts
```

职责边界：

- loader 只处理版本化资产和诊断，不访问 React state。
- playback math 是无 DOM 纯函数，处理 elapsed time、帧、循环和跨帧事件。
- `SvgRuntimeRig` 持有 DOM 引用，只负责投影与恢复，不决定动作何时开始结束。
- `SvgMotionPlayer` 持有一个播放 session、RAF 和 settle 逻辑，不操作调度队列。
- `SvgCharacterRenderer` 继续是业务层唯一渲染入口。

## 6. 资产加载状态

运行时加载应显式建模：

```ts
type MotionBundleState =
  | { status: "loading" }
  | { status: "ready"; bundle: ValidatedMotionBundle }
  | { status: "error"; diagnostics: RuntimeDiagnostic[] };
```

加载顺序：

1. 解析 rig/motions JSON。
2. `validateRig()`。
3. 计算 canonical artwork SHA-256，核对 source、fingerprint 和 viewBox。
4. `validateMotionLibrary(motions, rig)`。
5. 检查所有 `sourceBinding` 能在实际内联 SVG 中唯一命中。
6. 构建 runtime rig 后才把 capabilities 从空列表切换为 `motions.clips.map(id)`。

在 `loading/error` 状态调用动作必须返回 `renderer_unavailable`。缺失 Clip 必须返回 `unsupported_action`，不能回退成 CSS 动作或伪装 completed。

## 7. SVG Part 组合模型

### 7.1 包装层顺序

每个可动画 Part 使用稳定包装层，任何一层都不能覆写另一层的 transform：

```text
render-slot container
  interaction wrapper       # 视线、头部跟随、用户交互
    authored wrapper         # motion JSON：matrix + opacity
      procedural wrapper     # 呼吸、待机微动、头发、耳朵
        source Part          # 原始 SVG 几何
```

现有 CSS 中直接写在 `#body`、`#head`、`#arm-*` 和头发 source 上的 transform/animation，应迁移到对应 wrapper。运动播放器只写 `authored wrapper` 的：

```ts
transform="matrix(a b c d e f)"
opacity="..."
```

它不写 source Part 的 style，不清空程序动画 CSS 变量。

### 7.2 bind 与 pivot

- authored matrix 使用共享核心 `composeAroundPivot()`。
- DOM 投影矩阵为 `bindMatrix × authoredMatrix`。
- 若 source 原有 `transform` 已由 `bindMatrix` 表示，构建 wrapper 时把该 attribute 转移到 authored wrapper，并在 source 上清除；cleanup 时精确恢复。
- 不使用 `transform-origin` 百分比估算，不依赖 `getBBox()` 中心。
- 当前小洛宝 rig 是扁平绝对坐标；若运行时发现 source DOM 层级与 rig 组合假设不一致，初始化失败并给出 Part ID，不做隐式重定向。

### 7.3 render slot

- 在 `character` 下建立按 rig 顺序排列的 `back/body/head/front` 容器。
- Part wrapper 在同一 SVG DOM 中移动，不 clone、不 React 重挂载，因此 CSS 变量、WA animation 和事件引用持续有效。
- 每帧仅在离散 slot 发生变化时重排；同 slot 内按原始 source order 稳定排序。
- Clip 完成、中断、失败或 dispose 后恢复 default slot 和原始 DOM 顺序。
- 初始化测试必须证明正式素材的可重排 Part 坐标不会因换父级变化；不满足时改为矩阵补偿，不能接受视觉漂移。

## 8. 播放生命周期

### 8.1 session

播放器同时只允许一个 body motion session：

```ts
interface MotionPlaybackSession {
  id: number;
  clip: MotionClipV1;
  startedAt: number;
  speed: number;
  rafId: number | null;
  settled: boolean;
  lastAbsoluteFrame: number;
  resolve(): void;
  reject(error: RendererError): void;
}
```

统一 `settle(reason)` 必须：

1. 只执行一次。
2. 取消 RAF 和事件计时。
3. 清除动作事件覆盖层。
4. 恢复 default render slot、identity authored pose 和程序动画 suppression。
5. 移除 AbortSignal listener。
6. 根据完成或中断结算 Promise。

### 8.2 帧推进

- 使用 RAF timestamp，不使用 interval 累加。
- `elapsedFrames = elapsedMs / 1000 × fps × speed`。
- `loop: none` 必须精确采样一次 `durationFrames` 后完成。
- `loop: repeat` 使用绝对帧记录跨圈事件，持续到 AbortSignal、reset 或调度超时。
- speed 必须为已验证的有限正数；沿用协议当前允许范围。
- 页面停顿后按真实 elapsed time 恢复，不补跑每一个丢失的视觉帧，但必须正确收集跨过的白名单事件。

### 8.3 中断和错误

扩展 `MotionOptions` 以携带 `signal`；执行器把 scheduler 的 AbortSignal 直接传给 renderer。`reset("interrupt")` 仍作为统一恢复入口，但不能留下永不结算的底层 Promise。

建议增加带稳定 code 的 `RendererError`：

```ts
type RendererErrorCode = "unsupported_action" | "renderer_unavailable" | "invalid_asset";
```

`PetActionExecutor` 将已知 renderer error 映射为结构化 rejected/failed 结果；真正 DOM 异常才是 failed。调度器事件必须能区分 completed、interrupted、timed_out 和 failed。

## 9. 程序动画叠加与抑制

`suppressProceduralChannels` 只在当前 Clip session 内生效：

| 通道 | 需要控制的现有行为 |
|---|---|
| `breathing` | 根部呼吸、左右手待机微动、领带待机摆动 |
| `blinking` | 随机眨眼；动作事件触发的 blink 不受影响 |
| `pointer-follow` | 眼睛、眉毛、嘴、腮红、头、身体、手臂、马尾视线跟随 |
| `hair-physics` | 马尾/刘海/鬓发/发饰待机摆动和拖动弹簧 |
| `ear-twitch` | 双耳随机 Web Animation |

实现方式：

- `TianyiPet` 维护当前 suppression 集合，并传给 hooks/artwork。
- hooks 增加明确的 `enabled` 输入；关闭时取消 RAF、timeout、WA animation 并把自有 CSS 变量归零。
- CSS 通过 `data-suppress-*` 禁用对应 animation；不依赖 JS 修改 stylesheet。
- 玩家保存并恢复上一 suppression 集合，不能一律恢复为空，以免覆盖设置或其他上层状态。
- 拖动开始会中断 body motion；中断恢复后 `useHairMotion` 再开始采样，不能保留动作前的旧速度。

表情需要区分基础表达与动作事件覆盖：

```text
rendered expression = motion event overlay ?? base expression
```

这样 motion 内 blink/mouth 事件结束时不会把并行的 sleep/speak 状态错误恢复成 normal。

## 10. 动作事件白名单

P3 支持：

- `blink`：设置短暂动作表情覆盖。
- `mouthOpen` / `mouthClose`：设置或清除 mouth 覆盖。

P3 不执行：

- `sfx`：仅记录 `unsupported_event` 诊断，等待 M14 本地音频能力。
- `custom`：默认忽略并记录；只有未来显式注册的本地 handler 才可执行。

事件收集必须用纯函数覆盖：同帧、跨多帧、跨循环、起始帧、终止帧和一次 RAF 跨过多圈。任何 payload 都不得作为脚本、选择器、文件路径或网络 URL 执行。

## 11. reduced motion

检测 `prefers-reduced-motion: reduce` 时：

- 非循环动作不播放连续补间；采样安全终态后在下一 animation frame 恢复并 completed。
- repeat 动作不进入无限循环，立即 completed。
- 默认不执行 blink/mouth 之外的动态事件；仍不得执行 `sfx/custom`。
- 必须经过与正常动作相同的 settle/restore 路径。
- 切换系统偏好后下一次动作使用新策略；不要求中途改变正在播放的动作速度。

## 12. React 与 StrictMode 接入

- `TianyiArtwork` 通过 ref/callback 暴露一个小型 `MotionTarget`，不把 DOM Element 放进可序列化 React state。
- layout effect 构建 runtime rig；cleanup 恢复原 DOM、取消 player 并清除回调。
- `PetRuntimeProvider` 在 bundle ready 后刷新 capabilities，DebugConsole 不再持有初始化时的硬编码快照。
- HMR 重新载入 motions 只替换校验后的动作库并中断当前 session，不重建已稳定的 SVG rig。
- artwork 或 rig 改变才允许重建 rig；重建前先完整 cleanup，避免重复 wrapper 和 pivot 漂移。
- 所有 RAF、timeout、media query、AbortSignal 和 DOM listener 都必须可枚举并在 cleanup 释放。

### P3-5：真实 Tauri 回归与收口

通过门：编辑器制作的 `wave` 成为唯一生产挥手实现，桌宠既有窗口与程序动画行为无退化。

## 14. 测试计划

### 14.1 共享/纯函数

- elapsed time 到帧：0、末帧、speed、repeat、多圈和浮点边界。
- 跨帧事件：正好命中、跨多个、跨循环、跨多圈和不重复触发。
- transform/opacity 默认值与稀疏关键帧。
- renderer error 到 `ActionResult` 映射。

### 14.2 DOM/runtime rig

- 每个 rig Part 唯一命中 source binding。
- wrapper 顺序、bind 转移、matrix 与 cleanup 往返。
- opacity 与 interaction/procedural transform 同时存在。
- render slot 稳定排序、只在离散变化时移动、结束恢复。
- StrictMode 双挂载、重复初始化和 HMR 不产生重复 wrapper。

### 14.3 生命周期

- 正常完成、开始前 abort、播放中 abort、reset、dispose。
- repeat 直到 abort。
- renderer unavailable、未知 Clip、无 DOM target、损坏资产。
- reduced motion 立即完成且无残留。
- 旧 session 的 RAF 不得写入新 session。

### 14.4 集成

- `motion.play("wave")` 经过 validator、scheduler、executor、renderer 完成。
- 用户动作抢占 Agent body-motion。
- 用户拖动中断 wave，结束后恢复程序动画。
- Mock Agent 缺失动作返回 `unsupported_action`。
- capabilities 与动作库 Clip IDs 一致。
